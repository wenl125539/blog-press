路由的两种模式区别

(带着问题去理解)

  1. 为什么会出现前端路由。
  2. 前端路由解决了什么问题。
  3. 前端路由实现的原理是什么。
    1. 为什么会出现前端路由。

传统项目
为什么以前没有“前端路由”这个说法，又是怎么跳转页面呢?这里我称以前没有前端路由的项目为 ‘传统项目’

  • 传统项目整个项目都是 DOM 直出的页面，（SSR（next.js nuxt.js等）属于首屏直出，不属于传统项目）
    什么是DOM直出呢？简单说就是浏览器输入网址后去请求服务器，服务器直接返回一个HTML页面的效果就是DOM直出
    并且每次跳转都会重新请求HTML资源。所以整个页面都会重新刷新一遍
    • 博客园里面的就是服务器直接返回一个HTML页面的效果（直接看接口返回的数据）
    • 浏览器页面右键，点击 “显示网页源代码“ 就能看到整个HTML页面(网页上能看到什么图片或文字，你能在上述图片中找到相应的 HTML 结构，那也属于传统页面，也就是 DOM 直出。)

单页面
相对于传统项目，先在的 React 、 Vue 、 Angular 等著名单页面应用框架有一个共同的特点，便是“通过 JS 渲染页面”。

可以看出单页面应用 只有一个页面在展示，那如果我有十几个页面要互相跳转切换，咋整！！？？
这时候前端路由 应运而生，它的出现就是为了解决单页面网站，通过切换浏览器地址路径，来匹配相对应的页面组件。

    1. 再聊聊解决了什么问题

  • 从上图可以看出，SPA(单页面应用)，可以允许页面在不刷新的情况下更新了页面的内容，但也存在两个问题：

  1. 当用户刷新页面时，浏览器会默认根据当前URL对资源重新定位（回到最初的状态） - 这个动作不仅使得用户的前进后退操作无法被记录，而且它是不必要的，
    想想：SPA作为单页面，本身也只会有一个资源与之对应
  2. SPA对服务端而言，就是一个URL、一个资源。如何将其改变为“多个URL” 映射多个视图内容呢？
  • 那么前端路由是怎么解决一个资源多个URL映射呢
  1. 首先：拦截用户的刷新操作，避免服务端盲目响应，返回不符合预期的资源内容，把刷新这个动作完全放到前端逻辑里消化掉
  2. 感知URL的变化。前端给URL做些不会影响其本身性质的微小处理，不会影响服务器对它的识别，只有前端能感知到，进而通过JS去生成不同的内容
  • 怎么感知到URL的变化呢？这时候就用到了我们的主角 （hash模式和history模式）

hash模式

  1. hash就是指url尾巴后的#号以及后面的字符。这里的#和css里的#是一个意思。hash也被称为锚点，本身是用来做页面定位的，它可以是对应的id元素显示在可视区域内。
  2. 由于hash值变化不会导致游览器向服务器发出请求，而且hash改变会触发hashchange事件，hashChange事件中获取当前的hash值，并根据hash值来修改页面内容，则达到了前端路由的目的。游览器的进后退也能对其进行控制，所以人们在html5的history出现前，基本都是使用hash来实现前端路由的

优点

  • 游览器没有请求服务器，但是页面状态和url一一关联起来

缺点

  • 链接带#号 不美观
  • 要传递复杂的数据，会有体积的限制

history模式

  1. history模式给了前端足够的自由，hash的传参是基于url的，如果要传递复杂的数据，会有体积的限制，而history模式不仅可以在url里放参数，还可以将数据存放在一个特定的对象中。
  2. 在hash模式下，前端路由修改的是#中的信息，而游览器请求时是不带它玩的，所以没有问题。但是在history下，你可以自由的修改path，当刷新时，如果服务器中没有相应的响应或资源，会分分钟刷出一个404来。

优点

  • 链接不带#号
  • 可以传递复杂数据
  • 可额外设置title属性供后续使用；

缺点

  • history模式需要后端配合将所有访问都指向index.html，否则用户刷新页面，会导致404错误。

eg

location /test {
		  alias   html/project/文件目录;
			try_files $uri $uri/  /test/index.html;
}

  1. hash模式带#号比较丑，history模式比较优雅；
  2. pushState设置的新的URL可以是与当前URL同源的任意URL；而hash只可修改#后面的部分，故只可设置与当前同文档的URL；

  4. pushState通过stateObject可以添加任意类型的数据到记录中；而hash只可添加短字符串；
  5. pushState可额外设置title属性供后续使用；
  6. hash兼容IE8以上，history兼容IE10以上；
